3 Automation ~10 min

Fix and extend a flaky script

Why this matters at Tepia: we run Playwright (JS/TS) for web — you'll live in this daily. The everyday task isn't building frameworks, it's spotting why a test is flaky and fixing it properly. Mobile/Appium comes next, in Part 4.

This Playwright script logs into saucedemo.com and adds one item to the cart. It works… sometimes. It has one classic reliability bug.

await page.goto("https://www.saucedemo.com/");
await page.fill('#user-name', 'standard_user');
await page.fill('#password', 'secret_sauce');
await page.click('#login-button');
await page.waitForTimeout(5000);  // <-- the bug is here
await page.click('#add-to-cart-sauce-labs-backpack');
Your task — 3 things:
  1. Replace the arbitrary wait with a proper one (Playwright's auto-waiting / a web-first assertion), and briefly say why yours is better.
  2. Add an assertion confirming the cart badge shows 1.
  3. Add a second, separate check: attempt login as locked_out_user / secret_sauce and assert the error message appears.
Answer in Playwright with JavaScript or TypeScript — that's our stack.
Optional — skip if short on time: in 1–2 lines, how would you structure this in a real project? (Page Objects, fixtures, CI…)
4 Appium · mobile automation ~10 min

Review a mobile automation script

Why this matters at Tepia: mobile is what we build. We automate our iOS/Android apps with Appium, and mobile locators break far more easily than web ones. You don't need a device for this — we want to see if you can read a script and spot what will fail in the real world.
Context: below is a real-world-style Appium + WebdriverIO test for the login screen of an Android app. It passes on the developer's machine and fails constantly in CI and on other devices. Nothing here is a syntax error — every problem is a reliability or practice problem. 
// Appium + WebdriverIO — login flow (Android)
await driver.pause(8000);

const user = await driver.$('//android.widget.EditText[@index="0"]');
await user.setValue('qa@tepia.co');

const pass = await driver.$('//android.widget.EditText[@index="1"]');
await pass.setValue('Test1234');

await driver.$('//android.widget.Button[@text="LOG IN"]').click();

await driver.pause(5000);
const welcome = await driver.$('//android.widget.TextView[@index="3"]');
console.log(welcome.getText());

A1 · What's wrong here?

List the problems you can find, and for each one say why it breaks in CI or on a different device. There are several — find as many as you can.

A2 · Rewrite it

Rewrite the script the way you'd actually commit it. Assume the developers can add whatever accessibility ids you ask them for.

A3 · One platform, two apps

The same login test now has to run on iOS as well as Android. What changes, and what would you do so you're not maintaining two separate test suites?

Why this matters at Tepia: every mobile app we ship talks to a backend. Catching a broken response in Postman is far cheaper than catching it as a blank screen in the app.
Setup: the endpoint GET https://jsonplaceholder.typicode.com/users/2 returns a JSON user object with fields like id, name, email, and nested objects address.city and company.name.
